# ТЕХНИЧЕСКОЕ РУКОВОДСТВО И ДОКУМЕНТАЦИЯ ПРОЕКТА
## Веб-интерфейс фильтра безопасности (Security Filter App)
## 1. Назначение программы и общие сведения
Данное программное обеспечение представляет собой изолированный защитный веб-контейнер (превентивный контур защиты «Model Guard»), разработанный на базе веб-фреймворка Django и языка программирования Python 3.9+. Основное предназначение системы — обеспечение надежной изоляции процессов обучения и инференса моделей машинного обучения (ML) от деструктивного внешнего и внутреннего воздействия.
Приложение выступает интеллектуальным шлюзом (прокси-фильтром) между интерфейсом конечного пользователя (оператора ИС) и ядром ML-модели (например, прогностической модели Predictive Maintenance для телеметрии нефтегазового оборудования). Оно осуществляет глубокий входной контроль, фильтрацию деструктивных векторов атак, динамическое разграничение прав доступа сотрудников на основе ациклических графов (DAG) и визуализацию аналитических данных потока запросов.
## 2. Общая логика работы и алгоритмы защиты
Процесс обработки каждого входящего запроса состоит из следующих последовательных этапов:
-	Входной сигнатурный контроль: Перехват входящего текстового промпта оператора и его проверка на наличие сигнатур известных атак (SQL-инъекции, XSS-сценарии, попытки удаленного выполнения кода).
-	Динамическая проверка прав (DAG): Сопоставление идентификатора и роли текущего пользователя со структурой направленного ациклического графа, развернутого в базе данных, для пресечения действий внутренних нарушителей.
-	Криптографическая обработка: В случае успешного прохождения проверок (статус ALLOWED), данные транслируются в безопасный формат Base64 с добавлением системной соли (переменная окружения SECRET_KEY_ML) перед отправкой в ядро модели.
-	Блокировка и журналирование: При обнаружении угрозы запрос блокируется (статус BLOCKED), инцидент фиксируется в журнале безопасности для последующего аудита.
-	Агрегация и визуализация: Передача статистических данных во фронтенд для перестроения интерактивных графиков распределения запросов.
## 3. Назначение основных модулей и структура кода
Архитектура приложения соответствует паттерну Model-View-Template (MVT) фреймворка Django и обеспечивает строгое разделение ответственности (Separation of Concerns):
-	manage.py: Стандартный интерфейс командной строки (CLI) для администрирования Django-проекта, выполнения миграций и запуска сервера.
-	views.py: Ядро логики безопасности приложения. Содержит функцию validate_data(data) для сигнатурного анализа по черному списку (DROP, DELETE, exec(, script, alert(, def, import, eval()), функцию encrypt_data(data) для кодирования Base64 с солью, а также контроллеры агрегации статистики запросов.
-	models.py: Декларация реляционной структуры таблиц локальной базы данных с помощью Django ORM. Описывает сущности пользователей, ролей и ребер направленного ациклического графа (DAG) для контроля доступа.
-	templates/index.html: Фронтенд-компонент, рендерящий интерфейс оператора: форму ввода промпта, блоки динамического вывода статусов безопасности, результирующей base64-строки и аналитических диаграмм.
-	tests.py: Модуль автоматизированного тестирования на базе PyTest, содержащий тест-кейсы для проверки граничных условий и валидации алгоритмов фильтрации на легитимных и деструктивных выборках.
## 4. Инфраструктура развертывания
Приложение спроектировано с учетом принципов контейнеризации и инфраструктуры как кода (IaC), что гарантирует полную изолированность защитных механизмов от целевой операционной системы:
-	Контейнеризация (Docker): Для сборки приложения используется изолированный Docker-образ на базе легковесного дистрибутива Python. Контейнеризация исключает конфликты зависимостей и обеспечивает неизменяемость среды выполнения (Immutable Infrastructure). Контейнер изолирует переменные окружения, включая секретную соль SECRET_KEY_ML.
-	Автоматизация конфигураций (Ansible): Развертывание контейнеров, подготовка хостовых систем, обновление черных списков стоп-слов сигнатурного анализа и конфигурация сетевых доступов (Firewall/iptables) автоматизированы с помощью Ansible-плейбуков. Это обеспечивает повторяемость среды разработки, тестирования и промышленной эксплуатации (Production).
## 5. Мониторинг и оценка качества
Система непрерывного контроля и мониторинга развернута для превентивного выявления аномалий в потоке запросов и отслеживания производительности защитного контура:
-	Сбор метрик (Prometheus): Специализированные эндпоинты приложения отдают телеметрию в формате Prometheus. Осуществляется непрерывный сбор следующих ключевых метрик: суммарное количество обработанных промптов, динамическое соотношение разрешенных и заблокированных запросов (метрики ALLOWED / BLOCKED), время обработки одного запроса модулем сигнатурного анализа, а также загрузка процессора и оперативной памяти контейнера.
-	Визуализация и аналитика (Matplotlib & Панели): Внутренний модуль аналитики (в views.py) использует библиотеки Pandas и Matplotlib для динамического формирования графиков распределения трафика непосредственно внутри сессии пользователя. Графики рендерятся в потоковые байты (io.BytesIO) и передаются в HTML в виде base64-изображений, исключая необходимость замусоривания диска временными файлами.
## 6. Управление данными и персистентность
Надежность хранения конфигураций и информации об инцидентах достигается за счет разделения логических слоев и использования легковесных, но отказоустойчивых хранилищ:
-	Постоянное хранилище (Persistency): Для хранения учетных записей, ролевых моделей и конфигурации направленного ациклического графа (DAG) используется СУБД SQLite. Использование постоянных томов (Persistent Volumes / Docker Volumes) гарантирует, что при перезапуске, обновлении или ротации контейнера вся структура прав доступа и журналы безопасности полностью сохраняются.
-	Разграничение доступа внутренних нарушителей: Благодаря хранению связей и иерархий должностей в виде DAG-структуры в БД, система блокирует попытки горизонтального и вертикального повышения привилегий, изолируя скомпрометированные учетные записи внутренних сотрудников до выяснения обстоятельств.
